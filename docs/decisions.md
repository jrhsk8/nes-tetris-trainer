### 2026-06-23 — Stronger replay protection, tuck-seeking lateral, admin login + email allowlist, results stats + star ratings (grill-with-docs #7)

A seventh `/grill-with-docs` session: five owner reports/asks, resolved below and filed as **#74–#80**. **This round is entirely additive and play-side** — no StackRabbit, no bank regeneration, no new puzzle IDs, no orphaned attempts. It touches the play app, the auth layer, one core input function, and adds a single new table (star ratings) plus an email-allowlist table; the existing `attempts`/`puzzles` data is untouched. Deploy stays manual.

**Replay protection (#report 1: "I keep seeing the same puzzles") — #74, #75**

- **Persistent 200-window, derived from `attempts`.** The session-only in-memory ring of the last 10 ids is replaced by a sliding window of the **200 most-recently-attempted distinct puzzles**, computed from the `attempts` table (no new schema). It persists per-device for free (the anonymous session lives in localStorage) and becomes **cross-device once account linking lands (#77)**. Matchmaking keeps **band-widen-before-relax**: it widens the rating band until it finds an *unseen* puzzle and relaxes the window (permitting a repeat) only when the in-reach bank is exhausted. Chosen over permanently excluding every seen puzzle (the owner picked a window so the finite bank still cycles, oldest-first) and over a localStorage/DB ring (redundant with `attempts`, which already records every play). Size fixed at 200 (~20% of the ~1003 bank); a dynamic fraction was rejected as unnecessary now.
- **Miss replay — both an explicit mode and gentle auto-injection.** A **miss** is a puzzle attempted but never solved; it leaves the miss set once solved. Two paths bring misses back: a deliberate **Review-misses** mode (serves unsolved puzzles oldest-first, bypassing the window *and* the rating band) and **auto-injection** at **~1 in 10** normal serves when a miss is *due* (has fallen out of the 200-window). Chosen over review-mode-only and auto-only (the owner wanted both), and the gentle 1/10 over backlog-first (which would stop normal play feeling new). **Terminology:** this "replay" is *re-attempting a missed puzzle* — distinct from the existing **Replay** (the combo animation on the feedback board); the UI calls it *Review misses*. Supersedes the [anti-repeat cooldown](glossary.md#anti-repeat-cooldown) (last-10, session-only).

**Right-side J tuck (#report 2: "can't tuck the J into the col-4/col-8 holes on puzzle 1374") — #76**

- **Lateral becomes tuck-seeking; the model was never the problem.** The BFS model and the input are **navigation-complete** — the tuck *is* reachable (soft-drop the open column, slide in at the pocket row). The defect is the **#68 free-lateral ride-up rule**: pressing toward a column where the piece doesn't fit at the *exact* current row **ejects it to the top of that column**, so tucking demands pixel-perfect soft-dropping and any mis-press throws the piece out. Fix: `moveToColumn` (hence `lateralMove` and the mobile drag) changes from *"fits at current row, else ride up to the top"* to **"move to the reachable position in the target column nearest the current row, preferring at-or-below (tuck in); ride up only when nothing at-or-below is reachable,"** picking from `reachableStates` so the superset/soundness invariant holds. Verified backward-compatible against the #68 lateral tests (wall ride-up → row 4, open-pocket tuck → row 16, empty slide, off-screen/full-column blocks all unchanged). Adds a **navigation-completeness property test** (a BFS over the *actual* input moves must reach every `enumerateResting` placement) and a **named regression from puzzle 1374's real board**. Chosen over keeping the ride-up + adding a separate tuck control, and over a 1374-only patch (the class persists). Refines the [ghost placement](glossary.md#ghost-placement) lateral rule set in #68.
- **Replay animates tucks from the stored `boardKey`; the "tuck-free" assumption was false.** The #76 lateral fix made the player *able* to tuck, but the **feedback replay** still mis-animated a tuck *optimal* (e.g. **puzzle 1534**): the 2026-06-20 note "the Hz-invariance gate guarantees every optimal placement is tuck-free" is **wrong** — the narrowed Hz gate (#40) only requires the optimal be *reachable*, so a genuine tuck optimal is stored and served. The generator drops the resting `row` when persisting `optimalLine`/`combos` (only `{rotation, col}`), so the replay hard-dropped the piece **onto the ledge** instead of into the pocket. *Grading is unaffected* — it matches the canonical `boardKey` (#42), which encodes the true tuck outcome. Fix is **play-side, no bank touch**: `resolveLineByOutcome` (core) recovers the exact resting rows by searching the (few) reachable resting rows of each ply that reproduce the stored `boardKey`, and `Feedback` pins them before `buildReplay`. The player's own confirmed line already carries its rows, so a tuck the player makes animates faithfully too. A test asserts **the animation's final resting cells equal what the puzzle says** (the stored outcome), not a hard drop. Chosen over storing `row` in the bank + a migration (recovery fixes all ~1000 legacy puzzles immediately with zero schema/deploy coordination); persisting `row` at generation remains an optional later cleanup.

**Admin (#report 3: "make jrhsk8@gmail.com an admin") — #77, #78**

- **Sign-in affordance + anon→account linking (preserve UID).** The `SignIn` screen (email/password + Google + Discord) only shows when anonymous sign-in *fails*, so no one can actually log in and OAuth is effectively absent. Add a "Sign in" affordance for anonymous users that **links** the chosen identity to the existing anon user, **keeping the same UID** so rating/attempts/prefs/seen-window/misses all carry over (Supabase manual identity linking). This also gives every signed-in player **cross-device sync**, which is what upgrades #74/#75 to cross-device. The owner additionally flagged that **Google and Discord connections must actually work** — so this includes fixing the OAuth `redirectTo` to include the GitHub Pages base path, plus a **supervised/manual** step to enable the providers in the Supabase dashboard. Chosen over fresh-account-on-sign-in (discards anon data) and an owner-only quick path.
- **Admin = the curator role, gated by a verified-email allowlist.** "Admin" is exactly the existing [curation](glossary.md#curation) powers (flag + soft-delete/cull), reachable now that login exists; the client surface renames *curator* → *admin*. The gate moves from the UID-keyed `curators` table to an **email allowlist** checked in RLS — `(auth.jwt() ->> 'email')` in the allowlist, requiring `email_verified` and a non-anonymous session — and **jrhsk8@gmail.com is seeded** into it. Chosen over UID-keyed (brittle across re-auth/providers; the ask is literally email-based), a hardcoded-email constant (needs a redeploy to change), and a broader super-admin tier (overkill for one dev).

**Results — community stats (#report 4) — #79**

- **Show the numeric puzzle rating + a live community-correct-%.** The puzzle's Glicko rating (already on the row, never shown) plus a **community-correct-% = solved / total attempts**, computed **live at results time** (a count query; the player's own just-finished attempt is included) and displayed **always, with the sample size — `X% (N)`** — so a tiny sample is self-evident. Chosen: solved/total over first-attempt-per-distinct-user (the owner picked the simplest definition); live over an offline-batch column (always fresh, including brand-new puzzles, since the offline tally is run manually and can lag); always-with-count over hiding below a threshold.

**Results — star rating (#report 5) — #80**

- **1-5 star "how fun" rating, revealed after you rate.** Stars express **quality / fun**, deliberately *not* difficulty (that is the rating + correct-%), and become the crowd substrate for the future auto-"interestingness" curation gate. **One rating per user per puzzle, anonymous allowed, changeable** (upsert, own-row RLS) in a new `puzzle_star_ratings` table. The community **average is hidden until the player submits**, then revealed live (`avg ★ (N)`) — chosen over always-show (anchors the subjective score) and admin-only (no social proof). Per-attempt ratings were rejected (one user could dominate an average).

### 2026-06-22 — Free lateral movement, mobile drag, fixed-board mobile layout, 4 difficulty bands + tetris cap, dev in-play curation (grill-with-docs #6)

A sixth `/grill-with-docs` session: five owner reports/asks, resolved below. Filed as **#68–#73** (free-lateral #68, mobile drag #69, mobile layout #70, difficulty bands + tetris cap #71, dev curation #72, larger additive bank #73). **Favorable sequencing this round:** reports 1–3 are **play-app-only** (no bank touch); report 4 is a **cheap offline re-band** (recomputed from each puzzle's stored [combo table](glossary.md#combo-table) — replay the placements, count cleared rows — so no StackRabbit, no new puzzle IDs); report 5 adds a **schema migration + a larger _additive_ generation run** (new puzzles appended; existing puzzles, attempts, and ratings preserved). So unlike #5 and the earlier overhauls there is **no destructive full re-bank and no orphaned attempts**. Deploy stays manual.

**Input — free lateral movement (#report 1: "L/R sometimes won't move")**

- **Lateral becomes free column selection.** Today `PlacementInput` gates left/right on the BFS reachable set *at the piece's current floating row*, so a press silently does nothing when a wall of stack blocks that row (after a soft-drop, or under a tall stack). New rule: a press moves the piece to the target column at the current row if it fits (this still covers sliding *into* an open pocket = a tuck); otherwise the piece **rides up** to the highest row that fits in that column. Lateral is then never blocked unless the whole target column is full to the very top — the owner's "always moves unless it'd go off-screen." **Tucks/spins survive:** sliding into an open pocket is reachable at the current row (no ride-up), and reaching a tuck under a *solid* ledge is now an explicit **soft-drop-first** gesture (lateral stays collision-gated at depth). The superset binding invariant holds — every ride-up target is a BFS-reachable state. Chosen over keeping collision-gated lateral (fixing only the "stuck feel"), and over dropping player-side tucks entirely. Supersedes the [Ghost placement](glossary.md#ghost-placement) free-positioning sketch.

**Mobile — drag input (#report 2)**

- **Drag anywhere on the board to position.** The board is one control surface (no need to grab the few-cell piece): finger column → piece column, with the same free/ride-up rule. Depth (the rare tuck) uses the on-screen ▲/▼ soft-drop buttons, **not** vertical drag — keeping the gesture unambiguous and avoiding accidental tucks. Rotation via the existing on-screen buttons. **Commit via an explicit Confirm button**, not lift-to-place (which would drop on any stray touch). Chosen over grab-the-piece and tap-to-target; over full 2-D drag; over release-to-place.

**Mobile — fixed-board layout (#report 3)**

- **Board is a fixed anchor; a fixed bottom zone carries the rest.** Owner invariant: the board **never moves or resizes** between solving and feedback. Below it, a fixed-height zone holds the **controls** while solving and a **compact combo list + replay** while in feedback. The list is packed so the result + top combos show with **zero scroll**; deeper ranks sit behind a "more" expand (the owner flagged per-puzzle scrolling as clunky). Chrome collapses — nav/account behind a menu, rating to a one-line chip — and the [next box](glossary.md#next-box) stays visible in the residual space (top strip / bottom zone, wherever it fits without shrinking the board further). The board shrinks **only as much as needed** to fit board + next box + zone + slim header on one portrait screen with no page scroll ("shrink the board" was the owner's last resort, honored as a minimum). Chosen over keep-board-big-with-overlays and over require-landscape. Supersedes the 2026-06-20 "phones collapse to a stacked, scrollable column" line.

**Difficulty — 4 bands + tetris cap (#report 4)**

- **Add a `very-easy` band.** Four bands — `very-easy` / `easy` / `medium` / `hard` — bucketed by `acceptCount`; `very-easy` is the most forgiving (highest `acceptCount`), seeded below the current `EASY_SEED` (1300), extending the easy tail for new/low-rated players. Chosen over reserving `very-easy` for trivial patterns and over keeping 3 bands.
- **Tetris ⇒ capped at `easy`.** A puzzle where **any acceptable combo (score ≥ 97) clears a tetris** is capped at `easy` — never `medium`/`hard` — with `acceptCount` choosing `easy` vs `very-easy` under the cap, and the seed rating capped to match. Trigger is "any acceptable answer is a tetris" (the player can recognize-and-cash it to pass), chosen over "rank-1 optimal is a tetris" and over "board is tetris-ready". Mapping is a **ceiling** ("nothing harder than easy", the owner's words), chosen over force-`very-easy` and force-`easy`. *Definitional note to confirm at build:* "tetris" = a single 4-row clear by **one** of the two placements, not 2+2 split across both. Supersedes the 3-band model in `generator/src/pipeline/difficulty.ts`.

**Curation — dev in-play flag/delete (#report 5: "generate a large bank, cull the boring")**

- **Reframed as an in-situ, dev-only feature**, not a standalone batch-review tool (the owner's pivot). An **allowlisted curator** account — gated in **Supabase RLS, not the client** (delete mutates the shared bank), mirroring the `submissions` allowlist/own-row RLS pattern — gets two actions during normal play:
  - **Flag** — attach a free-text **comment**, appended to a new **append-only `puzzle_flags` log** `(id, puzzle_id, user_id, action ['flag'|'cull'], comment, created_at)`, so later review can mine patterns of *what makes puzzles boring*. Chosen over columns-on-the-puzzle-row (one overwriting comment, no history/author).
  - **Soft-delete (cull)** — write a `cull` log row (optional reason) and set the puzzle's new **`active boolean default true`** flag false; [matchmaking](glossary.md#matchmaking) filters `active = true`, so it leaves play immediately while the row, combos, and attempts survive (reversible). Chosen over hard-delete (irreversible, orphans attempts, discards the sweep+consensus cost).
- **Gating** is an **allowlist by auth UID/email** checked in RLS, chosen over an `is_admin` role column (overkill for one dev) and a client-only toggle (insecure). The play UI reveals the controls only to allowlisted accounts; the DB enforces it regardless of client.
- **Coverage is organic by choice.** The bank is grown large (an additive generation run), but a curator only reaches the puzzles matchmaking serves them — so flags/culls accrue best-effort over time, with **no systematic dev review mode**. Chosen over a large bank + review mode and over a normal-size bank. Accepted trade-off: most of a large bank never gets a curator's eyes; the flag log is the long-term substrate for a future auto-"interestingness" gate.
- **New schema:** `active boolean` on `puzzles` + a curator-scoped RLS **update** policy; a `puzzle_flags` table + curator-scoped RLS insert/select; matchmaking selection gains an `active = true` filter.

### 2026-06-22 — Letter grades, board-top result feedback, NES next box, keyboard loop, cleaner boards, upload hardening, rank-1 audit (grill-with-docs #5)

A fifth `/grill-with-docs` session: ten owner reports/asks, resolved into the decisions below. Several share one enabler — a **bank re-generation** — so they ship together (the re-bank also folds in decimal scores and the #55 consensus pass). To be filed as issues.

**Grading & rating**

- **A+ win line moves 95 → 97.** The graded Glicko outcome curve (`scoreToOutcome`, #51) keeps its shape but its neutral point moves to 97: ≥ 97 gains rating (convex up to 100 = full credit), < 97 docks toward the 0.10 floor. `CORRECT_SCORE_THRESHOLD` moves to 97 so "correct" and "A+" coincide. The owner wants "only A+ (97+) answers to score"; keeping the continuous curve was chosen over a hard binary win/loss and over an "only A+ updates the rating" rule (which would let a rating only ever climb — Glicko needs two-sided outcomes).
- **Scores display as letter grade + one-decimal number.** Everywhere a /100 showed (the verdict and every combo-list row), show a standard US 12-band letter plus the raw score to one decimal, e.g. `A+ 97.6`. Bands are half-open intervals on the real score: A+ = [97,100], A = [93,97), A- = [90,93), B+ = [87,90), B = [83,87), B- = [80,83), C+ = [77,80), C = [73,77), C- = [70,73), D = [60,70), F = [0,60) and unranked. The decimal makes `ComboEntry.score` a float (generator drops its `round()`; finer scores land in the re-bank, no separate migration). Credit phrases ("big gain") are dropped from the player view.

**Result feedback (#1)**

- **Grade banner on the board top + NES sound.** On result, a banner overlays the top of the board well with the big letter + decimal score — green for A+ (win), red for below (a non-A+ "A" still reads red) — persisting through feedback until the next puzzle. The right-rail verdict banner is slimmed to just the rating-change line (no duplicate grade). A distinct NES-style chiptune plays: an ascending arpeggio for A+, a soft neutral blip for below; mute toggle in Controls, default on.

**Play UI**

- **Remove the replay step counter (#3).** The `feedback-step` line under the board (`{label} · {step}/{n}`) is removed entirely — it was replay-animation playback progress, not a score; it was the misread "count out of 10."
- **NES next box (#4).** Give the next box a fixed-size recessed black box matching the board well's border/shadow, the piece centered at the board's cell scale (constant footprint, no per-piece resize), keeping a small NES-font "NEXT" caption.
- **Swap Next/Replay (#7).** "Next puzzle" moves to center under the board (primary action); "Replay" moves to the right rail.
- **Keyboard-only loop (#6).** Add rebindable `next-puzzle` (default `N`) and `replay` (default `R`) actions. `N` advances in feedback — deliberately not Enter/Space (which stay confirm) so the piece-2 confirm keypress cannot bleed into "next." On puzzle load, focus auto-lands on the board so the whole loop is no-mouse.

**Right-wall report (#5)**

- **No model bug; prove + guard + redeploy.** `PlacementInput` already gates every move and rotation against the reachable set, so a bar cannot reach a cell past col 9 while placing (flush-right at cols 6–9 is legal, not off-board). The report is most likely a stale deployed bundle or a misread legal flush-right placement. Add a property test asserting no reachable state for any piece/board renders an out-of-bounds cell, add a defensive render guard (Board never draws a cell outside the grid), then redeploy and retest.

**Generator — cleaner boards (#8)**

- **Strict-clean default + a variety lane.** The default geometric accept tightens to clean (target holes ≤ 1, bumpiness ≤ ~12, max height ≤ ~12); a small relaxed lane (~20% of the bank: holes ≤ 2, bumpiness ≤ ~20) preserves some texture for variety. Self-play `noiseRate` drops 0.12 → ~0.08 so candidates trend clean upstream. Fewer candidates per draw is accepted — cleaner boards are the better teaching material. Exact thresholds set from one calibration run. Requires the re-bank. (Considered and rejected: a hard single bar with no variety lane; and a full cleanliness-score quota system — more machinery than the goal needs.)

**Security — screenshot uploads (#9)**

- **Full upload hardening.** Today any session (including anonymous) can upload any file type/size to any path in the `submissions` bucket, unlimited times, with a client-chosen `image_path` and content-type ([data-access.ts](../packages/data/src/data-access.ts), [schema.sql](../packages/data/schema.sql)). Harden: bucket `allowed_mime_types` (png/jpeg) + `file_size_limit` (~5 MB) + private bucket; storage INSERT policy scoped to a per-user path prefix; server-generated path/uuid (drop client-chosen path & content-type); magic-byte validation; per-user pending quota; require a non-anonymous session to submit; and sandbox the offline parser (size / decompression-bomb limits, untrusted input) wherever it reads submissions.

**Puzzle 436 — rank-1 audit (#10)**

- **Diagnose the failure class, add a gate, let the re-bank fix it.** The owner is confident 436's rank-1 (bar-right) is worse than rank-2 (bar-left). Likely cause: an eval-only **well-preservation blind spot** (a moderate error below the deeper-confirm `rejectMargin` of 12), or a **mirror near-tie** the engine broke marginally — plausibly a legacy pre-#53/#55 puzzle. Plan: pull 436, re-evaluate both placements with deeper StackRabbit + BetaTetris; identify the failure class; add a generator gate/term that catches it (well-preservation and/or mirror-near-tie cull, or tighter deeper-confirm / BetaTetris value agreement); then the full re-bank excludes the whole class. No per-puzzle surgery (re-ranking individual puzzles and per-puzzle culling were both rejected in favour of the root-cause gate).

### 2026-06-22 — BetaTetris normal-net top-1 consensus is a standard generation gate (#55)

Builds on the #54 Phase-1 measurement below (now superseded as the *headline*): the owner settled the gate design in a 2026-06-21 grilling session and chose to make a BetaTetris consensus check **the usual puzzle-creation path going forward**, plus a one-time live-bank repair.

- **Agreement rule — normal net, exact top-1.** Keep a puzzle iff the BetaTetris **normal** net's #1 piece-1 policy move lands the SAME board as our stored optimal's piece-1 placement (matched by the canonical 200-char outcome key — convention-free; see `bt-bank-keys.ts`). This is **stricter** than the #54 *top-3, both-net* measurement; measured top-1 keep-rate on the live bank ≈ **61% (170/280)**.
- **Normal net only; the `perfect` net is dropped from the standard path.** It is trained for maxout/killscreen play (tetris-only, aggressive well management) — off-objective for a general stacking trainer. So the standard gate consults the normal net alone.
- **Post-generation filter, not an inline TS gate.** Generation stays TS/StackRabbit (`generate.ts`); the consensus check is a **final stage** that batches TS survivors and runs the Python BT-normal adapter (`betatetris-spike/consensus.py`) over them, dropping non-agreers and topping the cull back up. Reusable stage: `generator/src/pipeline/consensus.ts` (`filterByConsensus` + injectable `ConsensusJudge`; `betaTetrisJudge` is the production shell-out). BetaTetris stays **offline / generator-only** (the StackRabbit guardrail; GPLv3 — never shipped to `apps/play`).
- **Fail-closed.** A candidate BetaTetris cannot cleanly judge (engine error, unreachable outcome, odd-parity board, inject mismatch) is **rejected**, and the `bt-error` count is logged apart from genuine disagreement so flakiness never silently inflates the cull.
- **Filter, not re-rank.** Disagreers are **dropped**, never relabelled with BetaTetris's move — adopting BT's pick would orphan the StackRabbit-derived combo table and the graded-reward scores (#51) and break the "optimal is StackRabbit-confirmed" invariant.
- **Live-bank repair.** `generator/src/repair-bank.ts` self-backs-up the bank (`puzzles_bak_pre55_*`; no `*_bak_*` table is dropped), drops the top-1 disagreers, and backfills via generate→filter to restore each difficulty band's pre-repair count (#52 spread preserved). It asserts the shipped bank is **100% top-1-consensus**.

### 2026-06-22 — BetaTetris consensus keep-rate measured; spike's 0/33 was an artifact (#54 Phase 1)

**Superseded as the headline by the #55 entry above (2026-06-22).** The *both-net top-3 = 72%* figure was the Phase-1 *measurement*; the settled gate is the **normal-net exact top-1** rule (≈61% keep-rate), and the **perfect net is dropped** as off-objective. The measurement methodology and the "0/33 was an artifact" finding below remain valid and are what the top-1 gate is built on.


Phase 1 of #54 (MEASURE the keep-rate) is **done** — and it **overturns the *Consensus bank* premise below**. Full report: [FINDINGS-betatetris-consensus.md](../FINDINGS-betatetris-consensus.md). Tooling: `generator/src/bt-bank-keys.ts` (exports each puzzle's after-piece-1 outcome key in the production convention) + `betatetris-spike/keeprate.py` (reads BetaTetris's piece-1 policy, ranks our optimal in it). Measured over the whole 280-puzzle bank on both v1.0.0 nets.

- **Keep-rate is high and difficulty-uniform, not low-yield.** Both nets rank our stored optimal in their **top-3 policy for 202/280 = 72%** of puzzles; our optimal is a *reachable, considered* placement in **100%**. Keep-rate is flat across difficulty bands (~72% hard / med / easy) — consensus does **not** preferentially delete the tightest puzzles.
- **The spike's `0/33` was a measurement artifact, not engine disagreement.** `compare.py` called `InputPlacement` once per piece, but BetaTetris places in **two phases** (a forced pre-adjustment tap, then an `IsAdjMove` adjustment that actually locks). The single call never locked adj-moves, so it compared board0 / a mis-phased policy. Measuring the *adjustment-phase* policy with the correct cadence, and scoring by **policy rank/mass** (not single-best-outcome equality), is what reveals the real consensus.
- **Phase 2 is now an owner decision, deliberately not built autonomously.** The owner accepted a "low-yield / more-tactical / smaller bank" gate *on the basis of `0/33`*; that premise is falsified. A 72%, difficulty-uniform gate would cull ~28% of the bank without the intended tactical re-shaping. Reshaping the live bank on a now-falsified premise is a surface-don't-proceed call — the gate criterion (top-3-both 72% / top-1-both 42% / `π≥τ`) is held for the owner. The gate tooling is ready to consume whichever criterion is chosen.

### 2026-06-21 — BetaTetris baked into the sandcastle image (#54 now autonomous)

Amends the *Consensus bank* entry below: **#54 (BetaTetris true-consensus filter) is reclassified `supervised` → autonomous/sandcastle.** The only thing that made it supervised was provisioning — the offline BetaTetris build wasn't in the sandbox. That gap is the same one the image already closes for StackRabbit, and the spike's userspace-`micromamba` workaround was a *WSL-host* constraint (no sudo/compiler), **not** a Docker-image one (the image build runs as root with apt). So BetaTetris is now baked into `.sandcastle/Dockerfile` exactly like StackRabbit: a userspace micromamba env `bt` (python 3.12 + pytorch-cpu, the env validated 33/33 in the spike), the built `tetris` C++ extension (rotation, kR=4), and the v1.0.0 perfect/normal weights — all at image time so it is available **offline**. Invoked on demand (a Python forward pass), not a server, so no `onSandboxReady` start hook (just a non-fatal env smoke-check). The spike adapters (`betatetris-spike/{pull,compare,smoke}.py`) are parameterized by env (`BT_HOME`/`BT_REPO_PY`/`BT_MODELS`/`BT_OUT`, with WSL-fallback defaults) and now tracked; a `bt-run` wrapper runs them in the env.

- **Guardrail unchanged, reinforced.** BetaTetris stays strictly **offline / generator-only** — vendored only into the private, never-distributed sandcastle image, never linked into or shipped with the play app. This matters more than for StackRabbit because betatetris-tablebase is **GPLv3** (StackRabbit is MIT): GPL obligations trigger on *distribution*, which this build never does.
- **Costs (accepted).** Image grows ~1.2–1.5 GB (pytorch-cpu + ~166 MB weights) → ~4.3 GB, plus several minutes of one-time build. Requires an AVX2/BMI2 CPU (the host qualifies). The image must be **rebuilt** before #54 can run; until then the smoke-check fails and RALPH skips #54 with a blocker comment.
- **The verdict did *not* change.** The spike's NO-GO on a *blanket* BetaTetris gate still stands. #54 stays **Phase-1-first (measure the keep-rate)**, and a consensus bank is built only if that rate is workable. Going autonomous changes *who runs Phase 1* (RALPH, not a supervised session), not whether the gate is worth building.

### 2026-06-21 — Consensus bank: graded reward, difficulty-by-tightness, deeper-StackRabbit, BetaTetris consensus (grill-with-docs #4)

A fourth `/grill-with-docs` session, following the **BetaTetris cross-check spike** (`betatetris-spike/`). The spike framed the whole design: a strong second engine (the BetaTetris NN — CPU-runnable, no GPU) **never** picks a tower/holey line (corroborating #50) but matches our stored optimal **0/33** (0/33 even in top-30) — a *legitimate* objective difference (BetaTetris maximizes expected whole-game score with deep lookahead; our generator optimizes a static 2-ply board eval). So cross-AI agreement on the *single best move* is unachievable at scale, and a blanket BetaTetris accept/reject gate is **NO-GO** — worse, it would preferentially delete the **hardest/tightest** puzzles (on a tight puzzle BetaTetris's different-objective pick scores below our 95 bar). The owner's goal — *good moves still rewarded, puzzles correct, and the best unambiguously the best, with hard puzzles having few passing answers* — decomposes into four pieces, filed as issues #51–#54:

- **Graded reward curve (#51).** Replace the binary `solved ? 0 : 1` Glicko outcome with a quality-graded `scoreToOutcome(score)`: **gentle-above / steep-below** — 95 = neutral (0.5), convex up to 100 = full credit (97 ≈ 0.58), steeper linear penalty below 95 down to a 0.10 floor. Persists the player's numeric score per attempt (additive `attempts.score`). Rewards near-optimal, docks bad misses harder, caps the worst. RALPH-friendly.
- **Difficulty = enforced answer-set tightness (#52).** Difficulty already = `acceptCount` + `margin`; make it **structural** — bucket by measured `acceptCount` (hard = ≤ ~2 acceptable answers) and have generation deliberately span easy→hard, so a hard puzzle isn't "too easy to find anything that works."
- **Deeper-StackRabbit best-confirm (#53).** Confirm rank-1 with a deeper StackRabbit search (`playoutCount > 0`) vs today's eval-only value; re-rank/reject eval-only quirks. This is the **correct tool for "is this the best *by our objective*"** (a same-objective deeper search) — unlike BetaTetris, which answers a different question. Zero new infra; the spike's recommended fallback.
- **BetaTetris true-consensus (#54).** Not a blanket gate: keep only puzzles whose deeper-confirmed optimal is *also* high in BetaTetris's policy (both engines bless the answer) — **low-yield by design**, biases toward decisive/tactical positions; the owner accepted the smaller bank. **Phase 1 = measure the keep-rate first.** *(Originally supervised because the offline BetaTetris build wasn't provisioned; that build is now baked into the sandcastle image, so #54 is **autonomous** — see the **BetaTetris baked into the sandcastle image** entry above.)*

Spike verdict + reproduce details: `betatetris-spike/README.md` (the verdict + method are summarized in this entry; the standalone FINDINGS brief was retired once its durable content landed here and in the README). Regens for #52/#53 reshape which candidates survive — back up the bank first; deploy stays manual.

### 2026-06-21 — Combo score re-anchored to gap-from-best (#47, post-v2 QA)

Revises the 2026-06-20 "Combo-grading overhaul" **normalization + correctness anchor** (the two-piece model and `boardKey` outcome-matching are unchanged). The old score was a per-puzzle min-max normalization with the **worst legal combo** as the 0 anchor; that anchor is a deeply negative outlier (digging a hole / topping out — measured median worst-legal gap **~1084** eval units vs a rank-2 gap of **~4**), which stretched the bottom of the scale and compressed every reasonable move into the high 90s. A mediocre move scored ~96 and graded **correct**, and scores were not comparable across puzzles.

- **Margin chosen from data, not guessed.** `generator/src/gap-sample.ts` re-ran the sweep on 24 real bank puzzles and reported raw rank-1→rank-N gaps; a follow-up measured the eval cost of one buried hole. Findings: a clean-but-bumpier alternative costs **~4–8** eval units (median 4.3, p75 8.4); a hole-burying move costs **~12–22** (p25 12.2, median 21.6). `MARGIN = 8` sits in the gap between them — admits "within a little bumpiness of the best," rejects most hole-burying moves.
- **Gap-anchored display score (generator).** `score = clamp(round(100 − k·(bestValue − value)), 0, 100)`; the worst-legal `min` is dropped. `k = (100 − THRESHOLD)/MARGIN = 5/8 = 0.625`, pinned so the same absolute gap scores identically on every puzzle.
- **Correctness (core, unchanged surface).** `CORRECT_SCORE_THRESHOLD` stays 95; with the gap-anchored score, `correct = score ≥ 95` is now exactly equivalent to `gap ≤ MARGIN (8)`. `ComboResult` and `gradeCombo` (a `boardKey` lookup) are unchanged. A move the old min-max scored ≥ 95 but that is ≥ MARGIN below best now grades **incorrect** (e.g. a 20-unit gap: old ≈ 98 → new 88).
- **Bank regen** recomputes only `combos[].score` (and the difficulty signals derived from them); `boardKey`s and placements are unchanged. Backed up first; deploy stays manual. (Side effect: gap-anchored scores yield smaller `acceptCount`s, so seed ratings skew a little harder — acceptable; the difficulty mapping is tunable separately.)

### 2026-06-21 — Rank-1 outcome-quality gate (#50, post-v2 QA)

A player reported a tower/holey "optimal" ranked #1 over a cleaner line; a bank-wide audit confirmed StackRabbit's eval-only `rate-move` value occasionally crowns a degenerate board #1 on a minority of puzzles. Fixes, in the offline generator (`generator/src/pipeline/combo.ts`, `generate.ts`):

- **Outcome-quality gate (reject, don't re-pick).** After the combo sweep, the candidate is **rejected** when its value-best (rank-1) resulting board is *egregiously* bad relative to a swept alternative: `rank1-holey` (a no-taller alternative buries ≥ 3 fewer holes) or `rank1-tower` (rank-1 ≥ 12 tall when an alternative is ≥ 4 shorter). The whole puzzle is dropped rather than silently swapping the optimal.
- **Egregious-only, by design — the engine eval is trusted for shape.** A measured 43% of legitimate candidates have a value-best that is *mildly* dominated on raw max-height (e.g. the optimal leaves a height-4 board while a flatter height-2 line exists). The engine's eval weighs bumpiness, well shape, and tetris-readiness — things raw holes+height miss — so a couple of rows of height is a legitimate trade, **not** a bug. Only ~8% are egregious (the real tower/holey blunders). Rejecting all mild domination would gut yield and bias the bank toward "always flattest," removing strategically interesting puzzles, so the gate targets egregious cases only. The audit acceptance ("zero Pareto-dominated rank-1") is read as **zero egregiously-dominated** rank-1, matching the 13 deleted puzzles.
- **Value-sanity re-rank.** The stored top-K is re-ranked by a holes-dominance topological sort (`rankCombosBySanity`): a board with strictly **more holes AND no-lower height** can never outrank a cleaner one, regardless of engine value. Height-only differences keep the engine's order (its shape judgement). Past the gate, rank-1 is unchanged (the optimal stays the engine's value-best).
- **Re-tightened board-health floor.** A `maxStartHeight` (12) rejects near-topout start boards before the sweep — the tall board0s that produce the bad rank-1s in the first place.
- **Full bank regen** with the gate active (backed up first; deploy stays manual), restoring ~309 well-formed puzzles. Regen is additive: `boardKey`s and placements are unchanged in form; only which candidates survive + their `combos` change.

### 2026-06-21 — v2 overhaul (grilling session #3): tucks, difficulty, matchmaking, dedup, submission

A third `/grill-with-docs` session. **Correction (2026-06-21): the combo-grading overhaul #31–#35 from 2026-06-20 IS implemented and deployed.** origin/main (`56ef029`) carries the `combos` schema column, `packages/core/src/combo.ts`, `generator/src/pipeline/combo.ts`, the combo-threshold checker, and `ComboList.tsx`; the v1 `checker.ts` and value tables are gone. The live site runs this model. *(An earlier draft of this entry described the live code as still-v1 — that was written against a stale local clone six commits behind origin and was wrong.)* The remaining v1 surfaces are real, though: hard-drop-only placement enumeration *and* input (no tucks), no dedup, no difficulty metric, uniform-random selection, and player ratings that are computed but never persisted (the dev-bypass user can't satisfy RLS). v2 therefore **builds on the existing combo model and extends it**, shipping as **one effort with a single bank regeneration** (the owner chose one big release over a phased rollout). Decisions below supersede the PRD-v1 sections noted inline; where they touch combo code they *extend* it rather than rebuild it.

- **Combo-threshold grading (already built; extended here).** The two-piece combo model — always place both pieces; score 0–100 across the combo cross-product; **Correct = score ≥ 95**; no wrong-first short-circuit — landed in #31–#35 and is live. It already resolved the "my answer was identical to #1 but graded wrong" report (the v1 short-circuit had failed the puzzle on a first move that merely differed from the engine's *setup* move). v2 extends it with outcome-by-resulting-board matching (next bullet), tuck/spin combos, difficulty, and dedup — not a rewrite. Supersedes PRD § Checker.
- **Matching is by outcome, not placement tuple.** An attempt matches a stored combo iff it yields the **same resulting board (the locked cells after both pieces)** — not by `(rotation, col)`, not by input path. This is rotation-index- and path-independent, so app-vs-StackRabbit rotation-numbering mismatches can never mis-grade a same-cells answer, and tucks/spins are graded by where they rest. `placement.ts`'s `toPlacement` — which searches only the hard-drop placement space and returns `null` for any tuck — is **removed**; each stored combo carries its placements (for replay) plus a canonical resulting-board key (for matching).
- **Tucks and spins are first-class.** Reversing the PRD's "there is never a tuck answer": the generator enumerates **all collision-reachable resting placements** (hard-drop + tuck + spin) and StackRabbit values each, so a tuck/spin can be the optimal combo. *Binding invariant:* generation's enumerated placement set must be a **superset of what the play-app input can place**, or outcome-matching would wrongly reject a legal tuck. Supersedes PRD §§ Hz-invariance, Checker, and the Out-of-Scope "tuck/spin-required" line.
- **Hz-invariance kept but narrowed.** "Execution speed" now means **left/right traverse speed only** (slow-tap vs fast-DAS reaching distant columns). Tuck/spin *capability* is **granted** — assumed always executable — and is no longer gated; the gate keeps only puzzles whose optimal answer doesn't flip with horizontal movement speed. Expected to *raise* yield, since the old gate over-rejected tuck cases. (Operationalizing this against the live StackRabbit input-timeline API is a generation-tuning task: the timeline knob couples horizontal reach and tuck reachability, so isolating the former likely needs a geometric reachability pass plus an explicit horizontal-reach check, not a two-timeline comparison.)
- **Free-positioning input.** The ghost gains collision-aware move/rotate/**soft-drop** so the player can slide under overhangs and rotate into pockets — no timer, so it stays pure judgment. Replaces the hard-drop-only `legalColumns` model. Supersedes the PRD ghost-input sketch.
- **Difficulty is a generation property.** Computed per puzzle as a **combination of `acceptCount` (combos scoring ≥ 95) and `margin` (gap between the best combo and the best one below 95)** — few acceptable answers and/or a large gap = hard. Stored raw (both signals) and mapped to the puzzle's **seed rating** (harder → higher seed), which bootstraps matchmaking immediately instead of waiting for crowd data. Supersedes PRD § flat-seed / "difficulty not derived from engine signals."
- **Board-health floor relaxed.** The R3 floor kept boards StackRabbit rates *highly* — clean boards where many moves are fine — which biased the bank toward **easy**. It is relaxed to a **fairness/garbage-only** floor; the difficulty target shapes the rest, biased hard with an easy tail kept for low-rated/new players. Supersedes the 2026-06-20 board-health-floor decision.
- **Deduplication (new gate).** Reject a candidate whose **`(piece1, piece2)` match and whose board is within a small Hamming distance** of any puzzle already accepted (checked against the in-progress batch *and* the existing bank). Catches near-identical look-alikes, not just byte-identical duplicates. None existed before.
- **Rating persistence + matchmaking (fixes "rating never changes").** Root cause: the dev-bypass user is an all-zeros UUID with no auth session, so RLS `auth.uid() = user_id` fails and every write is silently swallowed by the `finish()` catch in `PuzzleSession.tsx`; the open-access live site has the same problem for everyone. Fixes: (a) **anonymous Supabase auth for every visitor**, giving a real `auth.uid()` so player ratings persist (per device, portable, linkable to email/Google/Discord later); (b) **puzzle ratings tallied offline in batches** from the `attempts` table (proper Glicko-2 rating periods) — the play app writes only `user_ratings` + `attempts`, puzzles stay read-only to the client; (c) **matchmaking selection** — draw the next puzzle at random from puzzles within a **rating band around the player** (auto-widening if too few), **excluding recently-seen puzzles** (a cooldown — a puzzle can return later, just not soon). One query delivers both "around my rating with variance" and "no repeats for a while." Supersedes PRD § random/sequential selection and the matchmaking Out-of-Scope line.
- **Screenshot submission (new).** A front-end affordance to submit puzzles, **screenshot-only to start**. Solving requires StackRabbit (offline-only, hard guardrail), so the browser only **uploads the image to a submission queue**; the **offline pipeline OCRs the NES grid into board + pieces, solves, runs gates/dedup/difficulty, and banks or rejects** it. Submitted puzzles go live after the next generation run; the engine is never deployed. Extends PRD § board-source pluggability (a screenshot source alongside self-play).
- **Accepted costs.** A **third full bank regeneration** (new puzzle IDs; post-regen attempts orphaned — same pattern accepted twice before). Schema: replace `first_values`/`second_values` with a `combos` top-K column (each combo's placements + 0–100 score + resulting-board key); add `accept_count` / `margin` difficulty columns; add a `submissions` table + a Storage bucket for images; add an RLS path for anonymous sessions. The **replay** must learn to animate tucks (it currently assumes every optimal placement is tuck-free and drops straight down). `checker.ts`, `placement.ts`, the placement enumeration, the input layer, and selection are all rewritten.

### 2026-06-20 — Combo-grading overhaul (grilling session #2)

A second `/grill-with-docs` session reworked grading, feedback, and board generation. These decisions **supersede parts of the earlier same-day UX-overhaul entry** — specifically the "no cross-product needed" premise, the exact-match checker, the strip-plot solutions chart, and the `first_values`/`second_values` model — and require **another full bank regeneration**.

- **Grading → two-piece combo model.** A puzzle becomes "find the best two-piece combo." The player **always places both pieces** (the wrong-first-move short-circuit in `checker.ts`/`PuzzleSession.tsx` is removed). An attempt is the combo `(placement₁, placement₂)`, scored by StackRabbit's evaluation of the board after both placements, field-normalized 0–100 across the puzzle's combos. **Correct = score ≥ 95** (within 5% of the best). The binary solved/failed signal for co-rating follows that bar (a near-miss and a catastrophic miss both count as failed — Glicko-2 stays binary).
- **No cross-product → cross-product, top-K stored.** This reverses the first overhaul's "a wrong first move ends the puzzle, so no cross-product is needed." Generation now sweeps the full cross-product (~400–1000 combos/puzzle) to rank and normalize, but **stores only the top-K (K ≈ 30, tunable)** ranked combos (placements + 0–100 score + total ranked count). Rows stay tiny; no live engine at play time. Player buckets: top-5 → highlighted in the list; ranks 6–K → shown with exact rank+score; beyond K → "too low to rank" (this is R5's "too bad to evaluate" case).
- **Gates.** The discrimination/unambiguity gate is **dropped** — combo-threshold grading needs no unique best. **Hz-invariance is kept**, retargeted from "optimal move per ply" to "the best combo is identical across slow-tap/fast-DAS." Puzzle quality now rests on board quality + the ≥95 bar (accepted risk: an occasional trivially-passable puzzle).
- **Board quality (R3) — StackRabbit board-health floor.** Keep a snapshot only if its **minimum best-move value across all 7 piece types** clears a moderate, tunable floor — a piece-independent proxy for "a board StackRabbit rates highly," since StackRabbit exposes no static board evaluation. A cheap geometric pre-filter (holes/bumpiness) drops obvious garbage first; the floor runs as an early gate before the combo sweep. Chosen over pure geometric thresholds because the owner wanted a StackRabbit-grounded measure; min-over-7 (not the actual current piece) keeps it about the board, not the piece draw; "don't overdo it" → keep the floor moderate to protect yield. Self-play noise may be lowered to feed cleaner candidates.
- **Feedback (R4/R5).** The strip-plot solutions chart is replaced by a **verdict banner** (Correct/Incorrect + 0–100 score) and a **ranked combo list**: top-5 with scores, the player's combo highlighted if among them, otherwise a row below showing rank+score (6–K) or "too low to rank" (beyond K). Rows are **interactive** — selecting one animates that combo on the central board (the existing replay parameterized by `(p1, p2)`); the player's move is selected by default.
- **Colors (R1).** The feedback/replay path never passed `colorGrid` to the board ([Feedback.tsx](../apps/play/src/feedback/Feedback.tsx), [replay.ts](../apps/play/src/feedback/replay.ts)), so the stack reverted to white after placement. Fix: make the replay **color-aware** (base = the puzzle's color grid; each dropped piece paints its NES color group) and pass `colorGrid` to the feedback board. The original stack never goes white; every animated piece (player's or engine's) carries its group color.
- **Layout (R6).** The board shipped at `min(40vh, …)` (not the ~80vh the first overhaul intended) under two stacked headers with `align-items: start`, forcing scroll. Fix: a **slim combined top bar** (small wordmark + Play/History/Controls nav, subtitle dropped), board grown to ~`min(72vh, …)` and the play area vertically centered, so it holds on one screen on desktop/tablet (phones still stack/scroll).
- **Accepted costs.** Another full bank regen → new puzzle IDs, the live bank replaced again, post-#27 attempts orphaned (same pattern accepted on the first overhaul). Schema: replace `first_values`/`second_values` with a `combos` column (top-K). `checker.ts` rewritten (no short-circuit; combo lookup + threshold).

### 2026-06-20 — UX overhaul (grilling session): layout, colors, solutions chart, replay, input, history

A `/grill-with-docs` session reworked the play app's UX. The decisions below supersede the relevant feedback/layout sketches in the PRD and are filed as phased GitHub issues (Phase 1 = client-only; Phase 2 = offline bank regeneration + schema; Phase 3 = client features that consume the new data).

- **Performance — drop the CRT layer.** Scroll/general jank traced to the full-viewport `body::after` overlay: `mix-blend-mode: multiply` plus a 160px-blur inset shadow forces a whole-viewport re-composite on every paint. Removing the overlay and the body grid texture for a flat look was chosen over keeping a cheapened effect. The 200-`div` board is kept as-is (a larger board is the same cell count).
- **Layout — flanking dashboard.** The board is the centered hero, sized to the viewport height (`min(~80vh, …)`) so it stays maximal with no scroll, and it never moves between phases. Supporting UI flanks it: left = rating + on-screen controls; right = the NES next-box while solving, then the solutions chart after an attempt (they never co-occur). One-screen is guaranteed on desktop/tablet only; phones collapse to a stacked, scrollable column. Every panel clips its children (`overflow: hidden` + `max-width: 100%`), which fixes the chart-overflow bug structurally. The optimal line animates on the central board in place.
- **Piece colors — authentic, via full regen.** The existing bank cannot recover per-cell colors (no piece history was stored), so authentic colors require a full bank regeneration with color-tracking self-play — chosen over client-side synthesis. Colors are stored as a *separate* 200-char color grid (`'0'` empty, `'1'/'2'/'3'` = NES color group) so the binary `Grid` and all metrics/checker logic stay untouched. Accepted cost: new puzzle IDs, the live bank replaced, existing attempts orphaned.
- **Next-piece box.** Rendered as a real piece graphic (spawn orientation, NES color) in a bordered box, top-right (NES-accurate). The current piece stays the on-board ghost (no separate display). The box is empty on placement 2 (no lookahead).
- **Replay animation.** The piece spawns at top-center and performs one eased rotate-and-slide during the upper part of the fall, then drops straight to rest; clip-checked so it never passes through the stack; a flash-and-collapse plays when a line clears; honors `prefers-reduced-motion`. Safe because the Hz-invariance gate guarantees every optimal placement is tuck-free.
- **Solutions chart — replaces the geometric-metrics table.** After an attempt, feedback shows two per-piece value distributions instead of the holes/bumpiness/height table. Because a wrong first move ends the puzzle, only two flat lists are needed (no placement1 × placement2 cross-product): piece-1 over all its legal placements, and piece-2 over all its legal placements on the board *after the optimal first move*. Both are precomputed at generation (`first_values` / `second_values`) — no live engine, sidestepping the deferred totalValue-% feature. Each is drawn as a strip plot (dots = alternatives, ★ = optimal, ● = the player's move) with a rank callout ("5th of 17"); positions are field-normalized 0–100 and rank is the headline number. The `boardMetrics` functions remain (the generator still uses them); only the player-facing table is removed.
- **Input — rebindable keys.** Defaults: ←/→ move, `z` = rotate CCW, `x` = rotate CW (this adds counter-clockwise rotation, which did not exist), Enter/Space confirm, ↑ = rotate-CW alias. Five actions are rebindable via a Controls panel (press-to-bind; conflicts warn rather than silently double-bind). Bindings persist in a new Supabase user-prefs table so they sync across devices, like the rating.
- **History view.** A header nav (Play / History / Controls) opens a sortable/filterable list of past attempts (date / difficulty / result), newest-first, paginated. Difficulty is read by joining attempts → puzzles. Each row re-opens the puzzle read-only in the Feedback view; only post-regen attempts are re-openable (older puzzle IDs are gone).
