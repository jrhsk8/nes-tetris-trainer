# Decisions

Lightweight log of decisions not captured in [PRD-v1.md](PRD-v1.md) (the PRD owns product/architecture decisions; this owns tooling, repo, and doc choices). Newest first.

### 2026-06-26 — Guests are real anonymous sessions, not a local-only fallback (anon sign-ins were disabled in prod)

Reverses the local-only-guest part of the 2026-06-24 "Continue as guest" note. The live error `getUserRating failed: invalid input syntax for type uuid: "guest-…"` traced to: the Supabase project has **anonymous sign-ins disabled** (verified live: `signInAnonymously` → `422 anonymous_provider_disabled`), so `continueAsGuest` fell back to fabricating a **local-only** user with a `guest-<uuid>` id — which is **not** a valid UUID, so every user-id-keyed DB call (`getUserRating`, the recent-window, misses, …) threw on the `uuid` cast. The fallback's `guest-` prefix was never used for detection (guests are identified by `isAnonymous`), so it bought nothing and broke everything. **Removed the fallback:** `continueAsGuest` now does a real anonymous sign-in and **throws** the provider error if anon is disabled, rather than fabricating an offline user (`apps/play/src/auth/auth.ts`; the now-dead manual `listeners` array went with it, since `onChange` already relays Supabase's `onAuthStateChange`).

**The real fix is operational, not code: enable Anonymous sign-ins in the Supabase dashboard (Authentication → Sign In).** This is also the linchpin for #39/#99: with anon disabled, signed-out visitors have no session, so their player ratings, attempts, and the live puzzle-rating drift (incl. the SECURITY DEFINER fix below) **persist nothing** — the "every visitor gets a real anonymous session" premise silently wasn't holding in prod. Enabling it makes the anon-session model, #99 live ratings, and the puzzle-rating write all real for casual visitors at once. Cost: a real `auth.users` row per visitor (the intended #39 model; add captcha/rate-limit if abused).

### 2026-06-26 — Live puzzle-rating write was RLS-blocked; fixed via a SECURITY DEFINER function

#99's live per-attempt puzzle-rating drift never landed for normal players. `updatePuzzleRating` did a direct `puzzles` UPDATE, but the only `puzzles` UPDATE policy is `puzzles_admin_update` (`is_admin()`) — so a non-admin (incl. anonymous) session matched **zero rows** and the write silently no-op'd (RLS filters the row out of the UPDATE; PostgREST returns success with 0 affected rows, so the `if (error) throw` never fired). Only the service-role offline tally actually moved puzzle ratings; the glossary's "(RLS no longer drops the write)" was aspirational. Fix: a SECURITY DEFINER `update_puzzle_rating(p_id, p_rating, p_deviation, p_volatility)` (schema.sql) that updates **only** the three rating columns past the admin-only policy (board/combos/tags/`active` stay protected), with the values clamped to wide sane bounds since it is now anon-reachable; `updatePuzzleRating` calls it via `rpc()` instead of a direct update. Mirrors the existing `puzzle_*_stats` SECURITY DEFINER pattern. Both the play app (anon/authed) and the service-role tally reach the write the same way now. **Requires applying `schema.sql` to prod** (idempotent) before/with the code deploy — until the function exists the rpc errors and the play app's best-effort catch swallows it (no worse than the prior silent no-op).

### 2026-06-26 — Backup snapshots self-prune

The destructive bank migrations (`reband` / `retag` / `additive-bank`) each snapshot the live `puzzles` bank to a dated `puzzles_bak_<date>...` table before mutating it. These were never cleaned up and had grown to 23 backup/scratch tables in Supabase (`puzzles_bak_*`, `attempts_bak_*`, `user_ratings_bak_*`, plus `puzzles_quarantine_*` / `puzzles_easy_keep` one-offs) — dropped by hand. To stop the reaccumulation, the three migrations' duplicated `backupBank` helpers were consolidated into `generator/src/bank-backup.ts`, which also exposes `pruneOldBackups(databaseUrl, keepDays = 14)`. After each backup the migration now prunes snapshot tables older than 14 days. Prune is conservative by construction: it only ever considers tables matching the `(puzzles|attempts|user_ratings)_bak_` prefixes **and** carrying a parseable 8-digit `YYYYMMDD` stamp — undatable one-offs (`*_bak_e41`) and the `quarantine`/`keep` tables are left untouched. Real schema tables can never match. Needs `psql` (already required for the backup DDL; present in the WSL/sandcastle run env, not on the Windows box).

### 2026-06-26 — Selection de-clustering, placement boost, live puzzle ratings (#99)

Owner feedback: ranked play "feels like the same puzzle types in a row," and both player and puzzle ratings "take too long to filter up/down" — they should move strongly early and settle after a number of rating changes. Diagnosed (the selection was uniform-random within a rating band, type-blind; ratings were throttled by the compressed outcome curve and an RD-blind feeder, and **puzzle ratings did not move live at all** — only the offline tally moved them). Researched how others do strong-early-then-settle (Glicko-2 RD, Lichess/Chess.com provisional, USCF/FIDE dynamic-K). Three changes:

- **Anti-streak de-clustering of selection.** Matchmaking stays random within the rating band, but the final pick is now **weighted to steer away from a run of the same type**: each of the last `ANTISTREAK_WINDOW` (5) serves that shares a candidate's **headline type** multiplies its pick weight by `ANTISTREAK_PENALTY` (0.35). Type = `dominantTag(tags)` (new in `@trainer/core`): the single most-salient tag a puzzle reads as (maneuver > clear goal > stacking shape > avoid-dependency; `'plain'` if untagged). Weights stay positive, so a band of one type still serves it (**never starves**) — it only spreads types out when alternatives exist. Pure, in `packages/data/src/matchmaking.ts`; the play loop tracks recent headline types in-session (`PuzzlePlay.tsx`). Chosen over a hard no-repeat-type rule (can widen/starve oddly when a band is one type) — keeps it random, just de-clustered. The streaks were a real statistical artifact of band composition (type correlates with difficulty), not a selection rule.
- **Placement boost (strong-early-then-settle).** A USCF dynamic-K idea layered on Glicko-2, riding on the subject's **RD** (no schema migration — RD already shrinks with games): the rating move is multiplied by `placementBoost(RD)`, ramping linearly from `PLACEMENT_BOOST_MAX` (3.0) at the seed RD (350) down to **1× at `SETTLED_DEVIATION` (110)** — the Lichess "provisional" boundary. Only the **rating delta** is amplified (RD/volatility keep their Glicko values, so the boost decays on its own); a single move is capped at `MAX_BOOSTED_DELTA` (160) so no fluke teleports a rating. Applies to **both player and puzzle** (new puzzles converge fast too). Constants tuned by simulation against `glicko2-lite`: a true-1900 player reaches ~1840 by **attempt 8** (vs ~41 attempts unboosted), then nudges in single digits once RD crosses 110. Did **not** touch the graded outcome curve (`NEUTRAL_SCORE` 97) — the owner chose the boost over loosening grading, so steady-state stays stable. In `packages/rating/src/glicko.ts` (`placementBoost`, `boostMove`, applied in `applyAttempt`).
- **Live puzzle ratings.** The play app now **persists the puzzle's drifted rating per attempt** (`updatePuzzleRating` in `record-attempt.ts`, best-effort so a puzzle-write hiccup never undoes the player update or the attempt row), so puzzles filter up/down in real time instead of only when the owner runs the offline `npm run tally`. Concurrency is last-write-wins read-modify-write (fine at this scale). **Tally interaction:** live per-attempt is now the authoritative puzzle path; the offline tally folds *all* attempts onto the *current* stored rating, so re-running it on top of live-updated ratings double-counts — treat `tally` as **superseded** for now (it would need a reset-to-seed-first rebuild to be a clean reconciler; deferred, no schema seed-column yet).

### 2026-06-25 — Implementation outcomes: per-piece maneuver + VITS generators (#91–#95)

Built and shipped the grill-#9 work below. Key findings that revised the plan:

- **#91 spin descends** + the input-reachability gate (`inputReachableRestingPlacements`/`isInputReachable` in `placement.ts`) shipped and verified live on 2443. The gate is reused by every maneuver generator to close the generator↔play loop.
- **The line-clear plan generalized only to T/J/L spins, not S/Z.** `constructForcedSpin` (carve the footprint out of two full rows + roof-search) forces reachable line-clearing spins for **T, J, L** (`forced-spin-bank-gen`, a piece-2 maneuver). **S/Z force ZERO** reachable line-clearing spins in any rotation (measured 0/650+ across random/constructed/self-play) — their line maneuver is a tuck, not a spin.
- **S/Z spins ship via the DIG insight (owner's idea), not line-clears.** A non-clearing clean S/Z spin is NEVER StackRabbit rank-1 (StackRabbit always has a cleaner non-spin home). But a spin that **clears a line AND reduces holes (a dig)** on a nearly-clean board IS rank-1 — exactly like the accepted t-spin digs. `forced-sz-dig-bank-gen` ships engine-agreed S/Z spins this way.
- **BetaTetris yield is gated by move-position, not just quality.** BT's policy narrowly enumerates non-hard-drop **piece-1** moves and drops them as `unreachable`. Measured strict-consensus keep rates: maneuver-as-**piece-2** (J/L spins) ~54% ≫ maneuver-as-**piece-1** (S/Z dig) ~24% ≫ varied piece-1 ~8%. **Lesson: frame a maneuver as piece 2 for volume.**
- **VITS calibration (the relaxed bar barely matters).** _(SUPERSEDED 2026-06-26 — these VITS turned out to be burns, not clean tucks; see the burn problem above.)_ The vertical-I tuck cleans a roofed 4-deep pocket and leaves a clean tetris well (tetris-ready after, not before). Measured: VITS I-tucks land BetaTetris policy **rank-1 100%** (24/24) — far above the planned top-3 relaxed bar; StackRabbit rank-1 too. The strict consensus drops them only on piece-2 (O) follow-up disagreement (`disagree-p2`), which the relaxed bar correctly ignores. So the relaxed bar's real job for a **piece-1 maneuver** is "don't require the trailing setup piece to also be BT's pick," not "accept a weak maneuver."
- **Variety guard.** A read-only `bank-dedup-audit.ts` (exact + near-identical board clusters) surfaced that the first S/Z dig batch reused near-identical right-side-pocket boards (one 12-puzzle cluster). Fix: varied board sources (random height/roughness/pocket-position-any-side/well) + in-batch board-Hamming dedup (`varied-maneuver-gen`, `vits-bank-gen`).

### 2026-06-25 — Spin descends into pockets, hold-to-snap drop, ghost removed, per-piece special-maneuver generators + VITS (grill-with-docs #9)

A ninth `/grill-with-docs` session: one bug report and three asks. Reports 1–3 are **entirely play-app-side** (no StackRabbit, no bank regeneration, no schema change, no new puzzle IDs) — they touch `move-engine.ts`, `placement.ts` (one core helper), `PlacementInput.tsx`, and `Board.tsx`. Report 4 is **generator-only** (offline), additive to the bank. Deploy stays manual. To be filed as issues.

**Report 1 — "t-spin doesn't work on puzzle 2443" (the spin law is t-spin-hostile)**

- **Diagnosed, not guessed.** 2443 is `O+T`; its rank-1 line is a hard-drop O then a **T-spin** into a 1-wide/2-deep notch (`T` at `rot 2, row 18, col 5`). The T-spin target **is** a valid resting placement, **is** in the generator's `enumerateResting`, and **is** reachable by the interactive input BFS — so it is **neither a parity violation nor a navigation-completeness gap**. The defect is the **#8 spin law**: rotation snaps to the **nearest** reachable rotated state (`nearestReachableState`, "preferring at-or-below, nearest"), which is **height-preserving** by design (the "rotational twin" of the tuck-seeking lateral glide). The natural player flow — walk the T to col 5, drop to its resting row (15, on the ledge), press rotate — just **spins the T on the ledge** (it stays at row 15); the only way into the notch is the undiscoverable sequence **rotate → soft-drop → rotate**. A height-preserving rotate can never screw a piece *down* into a pocket, so t-spins (and all pocket-spins) are effectively broken.
- **Fix: spin descends into the pocket.** On rotate, snap to the **deepest** reachable at-or-below state at the new rotation (not the nearest), riding **up** only when nothing at-or-below is reachable (the floor case, unchanged). Pressing rotate now drops the piece as far into the pocket as that rotation allows; 2443 becomes pure **rotate, rotate** (each rotate screws it deeper). **Descent is unconditional** — rotating in an open column drops the piece to the floor of that column; `Up` lifts it back if needed. Chosen over a "pocket-only descent" rule (needs a fuzzy definition of *pocket*, less predictable, harder to test) and over keeping `nearest` + auto-soft-dropping after each rotate (same effect, more moving parts than changing the one preference). **Lateral stays height-preserving** (the #76 tuck-seeking glide is unchanged) — so spin and lateral are deliberately **no longer exact twins**: sliding sideways should preserve height (a free glide), but rotating should descend (screw into the pocket). This splits `nearestReachableState` into a nearest variant (lateral) and a deepest variant (spin). Add a **spin navigation-completeness property test** (a BFS over the *actual* input moves, including the new spin law, must reach every `enumerateResting` placement) and a **named 2443 regression** asserting `rotate, rotate` from the natural drop position lands the stored t-spin. Verified it is *not* merely a stale deploy — the repo already ships spin-anywhere (#8); the height-preserving law is genuinely t-spin-hostile even when deployed.

**Report 2 — "holding down should snap the piece all the way down after a short delay"**

- **Tap = one row; hold = snap.** A tap on the soft-drop key/button drops exactly one row (fine control preserved). **Holding past ~250 ms snaps the piece to its settle row** straight down the current column — tuck-aware, so it stops on top of an overhang and the player tucks sideways from there (identical to repeatedly soft-dropping). The **60 ms per-row auto-repeat (#89) is removed entirely** — one clean rule. `Up` still raises one row (to lift off the floor for a spin). Applies to both the keyboard soft-drop key and the on-screen ▼ button (a hold timer on each). Chosen over keeping the per-row repeat and only snapping after a longer hold (two stages, more complex), and over an accelerating "race-down" (fiddlier to tune, slower than a snap). **This reverses #8's "no hard-drop / nothing teleports" line** — but the snap is gated behind a *hold delay*, not a discrete button, so it stays closer to #8's hold-to-accelerate intent than to a hard-drop key. The delay (~250 ms) is tunable from feel.

**Report 3 — "get rid of the ghost piece entirely"**

- **Remove the faint landing projection; keep the one piloted outline.** "Ghost piece" here means the dim, colour-coded **landing projection** (`landingCells` in `Board.tsx`, `computeLanding` in `move-engine.ts`) that #89 re-introduced to show where the piece would rest — **not** the piloted outline (the glossary's *Ghost placement*, which *is* the input and cannot be removed). Delete the landing projection and its wiring; the single piloted outline remains (**hollow while floating, glow when resting**). This returns the input to **#8's one-outline model**, which #89 had partially walked back. It dovetails with reports 1–2: with no landing projection, *dropping* (now effortless via hold-to-snap) is how the player sees where the piece lands — exactly the #8 rationale. Play-app-side only; no effect on grading or the bank.

**Report 4 — "architect J/L/S/Z-spin, spintuck, more tuck, and VITS puzzle generators"**

- **The tags already exist; this is generation only.** Per-piece spin tags (`s-spin`/`z-spin`/`l-spin`/`j-spin`) and the whole `tagPuzzle` classifier are already built and wired (`packages/core/src/tags.ts`, `SPIN_TAG`). The gap is *generating puzzles that produce those tags*.
- **Approach: constructive, replicating the proven `spin-bank-gen.ts` recipe per piece.** Self-play almost never yields these maneuvers and the spins it does find disagree with BetaTetris (0/115); the prior **hand-constructed synthetic batch (#2332–2401) was deactivated for looking artificial**. The live t-spins (e.g. 2443) that *did* survive were built by `spin-bank-gen.ts`: `constructTSpinDouble` forces a board where a **line-clearing** T-spin is the only way to complete two rows → full `assemblePuzzle` pipeline (real StackRabbit sweep + #50/#53 gates) → require the optimal to be the spin → dedup → **BetaTetris 7/7 consensus** → insert. **The line-clear is the crux**: a marginal eval-best spin is an objective quirk BetaTetris rejects, but a *line-cashing* spin-double is unambiguously the best play, so both engines agree **and** it does not read as artificial. So the new generators are per-piece **forced line-clearing-maneuver** constructors on the same skeleton.
- **Same quality bar for spins / spintuck / tucks.** Each new puzzle's special maneuver must **clear ≥ 2 lines** (unambiguously best) **and** pass **StackRabbit rank-1 + BetaTetris 7/7 consensus + the new interactive-reachability gate** (the optimal maneuver must be reachable by the actual input under the report-1 deepest-spin law — closing the generator↔play loop so every shipped spin is executable). Chosen over relaxing consensus (risks the deactivated-batch look) and over accepting non-clearing positional spins (exactly the marginal quirks BT rejects). Lower yield per type is the accepted cost — it matches what already worked. (Build-time caveat: a 2-line clear may not be geometrically natural for every piece's spin; if a type can only yield a single-line clear, surface it rather than silently shipping weaker puzzles.)
- **VITS (vertical I tuck setup) is a different shape with its own bar.** VITS is **not** a line-clearing maneuver. Its fixed two-piece structure: **piece 1 sets up a 4-deep roofed tuck pocket; piece 2 is a vertical I tucked into it cleanly** (no line clear, no new holes). Because it clears nothing, it cannot use the line-clear + 7/7-consensus bar (a non-clearing positional I-tuck is exactly the objective call BetaTetris rejects). VITS acceptance instead requires: **StackRabbit rank-1** for the clean I-tuck, **a tetris-ready payoff** (the I-tuck leaves the well a clean tetris setup — giving the "setup" a concrete purpose and making the tuck genuinely best), **interactive-reachability**, **and a *relaxed* BetaTetris check** — BT must still "agree at least a little," but **below** the exact-top-1 gate. The relaxed threshold is **not hardcoded**: first run a calibration measurement (the #54 `keeprate` tooling) to see where VITS I-tucks land in BT's policy-rank distribution, then set the bar from the data (as `MARGIN` was set in #47). Working default until measured: **top-3 in BT's policy** (#54 found our optimal there 72% of the time — meaningful but not spin-hostile).
- **Rollout: insert directly active.** All gates above run regardless; survivors go into the live bank **active**, to be culled later via the in-play [admin](glossary.md#admin) tools if any look bad. Chosen over dry-run-then-review and insert-inactive-then-flip — the owner accepted the cull-later risk for speed, since the gates are stricter than the ones that produced the deactivated batch. A modest first-batch target per type, tunable by yield.

### 2026-06-24 — Guest auth bypass, tuck/spin puzzle generation, type-tags, editorial-dark UI

**Guest auth (#continue-as-guest)**

- **"Continue as guest" login bypass.** When Supabase anonymous sign-ins are disabled (or fail), the login page now offers a **Continue as guest** button that creates a **local-only guest user** (`guest-<UUID>`) and fires the `onChange` listeners to bypass the auth gate. The guest user has `isAnonymous: true` and a `crypto.randomUUID()` id; their data is device-local (no Supabase session, so no RLS-gated writes persist). This unblocks play on deployments where anonymous auth is not configured. Added to `AuthApi` as `continueAsGuest()` (`apps/play/src/auth/auth.ts`); the button is styled with a dashed border at reduced opacity (`apps/play/src/styles.css`).

**Tuck/spin puzzle generation findings**

- **Tuck-aware self-play.** Standard self-play misses tucks because it follows the engine's hard-drop-only placement. When the engine recommends a tuck move (i.e. `toHardDropPlacement` returns null for the engine's placement), the generator now **applies the engine's resulting board directly** instead of falling back to a random move. This builds boards with natural overhangs that produce tuck opportunities organically. Implemented in `generator/src/tuck-gen.ts`.
- **Yield rates.** Tucks: **~1+ per self-play board** (common — most boards with overhangs have at least one tuck-optimal combo). Spins: **~1 per 20 boards** (rare — requires a pocket geometry that only rotation can reach). T-spins are the most common spin subtype.
- **Engine crash management.** StackRabbit's C++ engine segfaults on certain board states (tall/messy boards with extreme configurations). On Windows, orphaned engine processes open console windows. Mitigations: `killEngine()` with `taskkill /T /F` for process-tree cleanup; a `consecutiveCrashes` counter (max 5) that aborts the run if the engine is unstable; process exit handlers (`process.on('exit'/'SIGINT'/'SIGTERM', killEngine)`); removed `detached: true` and `unref()` from spawn to prevent orphaned processes.
- **`--spin-only` flag.** `tuck-gen.ts` supports `--spin-only` to filter out pure tucks and keep only spin/spintuck results, useful for targeted spin generation given their rarity.
- **30 tuck puzzles inserted** (#2402–#2431) in the live bank; 70 earlier synthetic tuck/spin puzzles (#2332–#2401) remain deactivated. Spin puzzle insertion is pending (the generator finds them but no completed run has inserted them yet).

**Puzzle type-tags (#81, #84, #90)**

- **Auto-computed per-puzzle tags** describing what the puzzle teaches, added at generation time. Four tag families: **clear** (burn, tetris, tetris-ready, dig), **maneuver** (tuck, spin, t-spin), **stack** (clean-stacking, well-maintenance), **avoid** (avoid-{i,s,z,j,l}-dependency). The [maneuver](glossary.md#maneuver) classifier (`packages/core/src/tags.ts`) uses a BFS to distinguish hard-drops from tucks from spins. Tags display as coloured chips in the play app and power drill-mode filtering.
- **T-spin tag.** A spin involving the T piece is additionally tagged `t-spin` — the most common spin type because the T's symmetric cross shape fits more pocket geometries.
- **Avoid-dependency tags.** When the optimal combo's resulting board avoids creating a single-piece dependency that non-optimal combos create, the puzzle is tagged `avoid-<piece>-dependency`. Dependencies: I (deep well), S/Z/J/L (notch only one piece fills cleanly).

**Editorial-dark UI redesign (v2)**

- Applied the v2 editorial-dark visual redesign across the play app: dark background, refined typography, consistent spacing, updated colour palette for board cells and UI chrome. Purely cosmetic — no gameplay or data changes.

### 2026-06-23 — One free-floating piece outline: collapse the active/ghost split, spin-anywhere, resting glow (grill-with-docs #8)

An eighth `/grill-with-docs` session: one owner report bundling three symptoms of the placement-input rendering — **(a)** a piece **can't spin when resting at the bottom**, **(b)** a piece one row off the bottom shows an **awkward partial ghost**, **(c)** the distinction between *ghost*, *piece at the top*, and *placement-ready-to-lock* is **unclear**. Root cause is the **#81 dual rendering**: a bright *active* piece at the floating `(row, col)` plus a muted *drop-shadow ghost* at where it would land — two renderings that diverge by a row (the "partial ghost") and read as several ambiguous states. The fix collapses them to a **single free-floating outline**. **Entirely play-app-side** — no StackRabbit, no bank regeneration, no schema change, no new puzzle IDs; it touches `PlacementInput`, `Board`, and adds one shared core helper. Deploy stays manual. To be filed as issues.

- **One free-floating outline replaces the active-piece + drop-shadow pair.** During placement there is exactly one piece representation: a **free-floating, colour-coded outline** the player pilots. It does **not** auto-fall — it stays exactly where piloted (a [free cursor](#ghost-placement), WYSIWYG). The separate muted drop-shadow is **removed**, which deletes the "partial ghost one-row-off-the-bottom" symptom *at the root* — there is no second rendering left to diverge from the piece. Chosen over keeping the pair and merely restyling it (the duality itself was the confusion the owner reported).
- **Spawn floating at the top row — no air rows.** The outline appears floating at the board's top row (row 0), not pre-settled and not locked, and the player brings it down. The owner explicitly pictured "at the top, floating," **not** in blank space above the top edge — so **no rows are drawn above the playfield** (which, given the no-scroll layout where cell size = board-width / 10, would have shrunk every cell). Chosen over an air band above the edge once the cost (smaller cells) was made explicit and the owner confirmed they only meant "starts at the top, un-settled."
- **Spin becomes the rotational twin of the [tuck-seeking lateral](#ghost-placement) law — fixes spin-at-the-bottom for every piece.** Today rotation only applies if the new orientation fits at the *exact* current row/col, so it silently no-ops on the floor. New rule: **hold the column fixed, change rotation, and snap to the reachable state at the new rotation nearest the current row, preferring at-or-below (drop into the pocket), riding up only when nothing at-or-below is reachable** (the floor case — the only room is up). This is `moveToColumn`'s exact law applied to rotation instead of column, drawing candidates from `reachableStates`, so it is generalized into one shared *"nearest reachable state, tuck-in-preferred"* helper. **Researched outcome — no per-piece kick table.** NES Tetris has **no SRS / no wall-or-floor kicks**; every real T/S/Z/L/J/I "spin" works by rotating *at a height where the rotated shape already fits* then settling, and `reachableStates` already enumerates exactly those. Deferring to that set makes all piece spins work uniformly **and preserves generator parity**; an SRS kick table would have been *wrong* here (it would grant placements the generator never enumerated). Spin stays **column-fixed** (matches NES and the BFS): landing a spin in an offset pocket is a spin-then-shift (two clear presses), chosen over column-kicking spins (blurs spin vs shift, feels like teleporting).
- **Confirm is gated on *resting*; the glow is the cue.** A free cursor can sit mid-air, but a lockable placement must be **resting** (`isResting` — fits and can't fall one row), since grading matches the generator's resting-placement outcome set. So **Confirm is enabled only when the outline is resting**, and the outline signals it: **hollow outline while floating, outline + a glow the moment it is resting/lockable.** One piece, two states tied to lock-readiness — directly answering "which state am I in?" Chosen over a button-only cue (the board itself gives no feedback) and over filling the outline solid (the owner wanted it to stay an outline). The feedback view's gold answer-highlight is a separate concern and is unchanged.
- **Soft-drop only, hold-to-repeat; no hard-drop.** With no drop-shadow, *dropping is also how you see where it lands.* Down soft-drops one row and **auto-repeats quickly while held** so dropping far is effortless; Up raises one row (needed to lift off the floor to spin); there is **no hard-drop / snap** button — nothing teleports, nothing is tedious. Lateral keeps the existing [tuck-seeking](#ghost-placement) `moveToColumn` (height-preserving, so sideways reads as a free glide). Chosen over a separate hard-drop button (the owner preferred hold-to-accelerate) and over step-only soft-drop (too many presses to a low floor).
- **Parity is safe for free.** Every move and spin snaps only to states already in `reachableStates`, and locking only ever happens at a **resting** position — exactly the set `enumerateResting` gives the generator. Nothing in this redesign expands the legal-placement universe, so the [superset binding invariant](glossary.md#tuck--spin) (generator↔play parity) holds unchanged. No bank touch, no `boardKey` change, no regrade.

### 2026-06-21 — Consensus bank: graded reward, difficulty-by-tightness, deeper-StackRabbit, BetaTetris consensus (grill-with-docs #4)

A fourth `/grill-with-docs` session, following the **BetaTetris cross-check spike** (`engines/betatetris/`). The spike framed the whole design: a strong second engine (the BetaTetris NN — CPU-runnable, no GPU) **never** picks a tower/holey line (corroborating #50) but matches our stored optimal **0/33** (0/33 even in top-30) — a *legitimate* objective difference (BetaTetris maximizes expected whole-game score with deep lookahead; our generator optimizes a static 2-ply board eval). So cross-AI agreement on the *single best move* is unachievable at scale, and a blanket BetaTetris accept/reject gate is **NO-GO** — worse, it would preferentially delete the **hardest/tightest** puzzles (on a tight puzzle BetaTetris's different-objective pick scores below our 95 bar). The owner's goal — *good moves still rewarded, puzzles correct, and the best unambiguously the best, with hard puzzles having few passing answers* — decomposes into four pieces, filed as issues #51–#54:

- **Graded reward curve (#51).** Replace the binary `solved ? 0 : 1` Glicko outcome with a quality-graded `scoreToOutcome(score)`: **gentle-above / steep-below** — 95 = neutral (0.5), convex up to 100 = full credit (97 ≈ 0.58), steeper linear penalty below 95 down to a 0.10 floor. Persists the player's numeric score per attempt (additive `attempts.score`). Rewards near-optimal, docks bad misses harder, caps the worst. RALPH-friendly.
- **Difficulty = enforced answer-set tightness (#52).** Difficulty already = `acceptCount` + `margin`; make it **structural** — bucket by measured `acceptCount` (hard = ≤ ~2 acceptable answers) and have generation deliberately span easy→hard, so a hard puzzle isn't "too easy to find anything that works."
- **Deeper-StackRabbit best-confirm (#53).** Confirm rank-1 with a deeper StackRabbit search (`playoutCount > 0`) vs today's eval-only value; re-rank/reject eval-only quirks. This is the **correct tool for "is this the best *by our objective*"** (a same-objective deeper search) — unlike BetaTetris, which answers a different question. Zero new infra; the spike's recommended fallback.
- **BetaTetris true-consensus (#54).** Not a blanket gate: keep only puzzles whose deeper-confirmed optimal is *also* high in BetaTetris's policy (both engines bless the answer) — **low-yield by design**, biases toward decisive/tactical positions; the owner accepted the smaller bank. **Phase 1 = measure the keep-rate first.** *(Originally supervised because the offline BetaTetris build wasn't provisioned; that build is now baked into the sandcastle image, so #54 is **autonomous** — see the **BetaTetris baked into the sandcastle image** entry above.)*

Spike verdict + reproduce details: `engines/betatetris/README.md` (the verdict + method are summarized in this entry; the standalone FINDINGS brief was retired once its durable content landed here and in the README). Regens for #52/#53 reshape which candidates survive — back up the bank first; deploy stays manual.

### 2026-06-21 — Combo score re-anchored to gap-from-best (#47, post-v2 QA)

Revises the 2026-06-20 "Combo-grading overhaul" **normalization + correctness anchor** (the two-piece model and `boardKey` outcome-matching are unchanged). The old score was a per-puzzle min-max normalization with the **worst legal combo** as the 0 anchor; that anchor is a deeply negative outlier (digging a hole / topping out — measured median worst-legal gap **~1084** eval units vs a rank-2 gap of **~4**), which stretched the bottom of the scale and compressed every reasonable move into the high 90s. A mediocre move scored ~96 and graded **correct**, and scores were not comparable across puzzles.

- **Margin chosen from data, not guessed.** `generator/src/gap-sample.ts` re-ran the sweep on 24 real bank puzzles and reported raw rank-1→rank-N gaps; a follow-up measured the eval cost of one buried hole. Findings: a clean-but-bumpier alternative costs **~4–8** eval units (median 4.3, p75 8.4); a hole-burying move costs **~12–22** (p25 12.2, median 21.6). `MARGIN = 8` sits in the gap between them — admits "within a little bumpiness of the best," rejects most hole-burying moves.
- **Gap-anchored display score (generator).** `score = clamp(round(100 − k·(bestValue − value)), 0, 100)`; the worst-legal `min` is dropped. `k = (100 − THRESHOLD)/MARGIN = 5/8 = 0.625`, pinned so the same absolute gap scores identically on every puzzle.
- **Correctness (core, unchanged surface).** `CORRECT_SCORE_THRESHOLD` stays 95; with the gap-anchored score, `correct = score ≥ 95` is now exactly equivalent to `gap ≤ MARGIN (8)`. `ComboResult` and `gradeCombo` (a `boardKey` lookup) are unchanged. A move the old min-max scored ≥ 95 but that is ≥ MARGIN below best now grades **incorrect** (e.g. a 20-unit gap: old ≈ 98 → new 88).
- **Bank regen** recomputes only `combos[].score` (and the difficulty signals derived from them); `boardKey`s and placements are unchanged. Backed up first; deploy stays manual. (Side effect: gap-anchored scores yield smaller `acceptCount`s, so seed ratings skew a little harder — acceptable; the difficulty mapping is tunable separately.)

### 2026-06-21 — Rank-1 outcome-quality gate (#50, post-v2 QA)

A player reported a tower/holey "optimal" ranked #1 over a cleaner line; a bank-wide audit confirmed StackRabbit's eval-only `rate-move` value occasionally crowns a degenerate board #1 on a minority of puzzles. Fixes, in the offline generator (`generator/src/pipeline/combo.ts`, `generate.ts`):

- **Outcome-quality gate (reject, don't re-pick).** After the combo sweep, the candidate is **rejected** when its value-best (rank-1) resulting board is *egregiously* bad relative to a swept alternative: `rank1-holey` (a no-taller alternative buries ≥ 3 fewer holes) or `rank1-tower` (rank-1 ≥ 12 tall when an alternative is ≥ 4 shorter). The whole puzzle is dropped rather than silently swapping the optimal.
- **Egregious-only, by design — the engine eval is trusted for shape.** A measured 43% of legitimate candidates have a value-best that is *mildly* dominated on raw max-height (e.g. the optimal leaves a height-4 board while a flatter height-2 line exists). The engine's eval weighs bumpiness, well shape, and tetris-readiness — things raw holes+height miss — so a couple of rows of height is a legitimate trade, **not** a bug. Only ~8% are egregious (the real tower/holey blunders). Rejecting all mild domination would gut yield and bias the bank toward "always flattest," removing strategically interesting puzzles, so the gate targets egregious cases only. The audit acceptance ("zero Pareto-dominated rank-1") is read as **zero egregiously-dominated** rank-1, matching the 13 deleted puzzles.
- **Value-sanity re-rank.** The stored top-K is re-ranked by a holes-dominance topological sort (`rankCombosBySanity`): a board with strictly **more holes AND no-lower height** can never outrank a cleaner one, regardless of engine value. Height-only differences keep the engine's order (its shape judgement). Past the gate, rank-1 is unchanged (the optimal stays the engine's value-best).
- **Re-tightened board-health floor.** A `maxStartHeight` (12) rejects near-topout start boards before the sweep — the tall board0s that produce the bad rank-1s in the first place.
- **Full bank regen** with the gate active (backed up first; deploy stays manual), restoring ~309 well-formed puzzles. Regen is additive: `boardKey`s and placements are unchanged in form; only which candidates survive + their `combos` change.

### 2026-06-21 — v2 overhaul (grilling session #3): tucks, difficulty, matchmaking, dedup, submission

A third `/grill-with-docs` session, building on the already-deployed combo-grading overhaul #31–#35 (the `combos` schema, the combo-threshold checker, and the combo-list UI; the v1 `checker.ts` and value tables are gone). The remaining v1 surfaces are real, though: hard-drop-only placement enumeration *and* input (no tucks), no dedup, no difficulty metric, uniform-random selection, and player ratings that are computed but never persisted (the dev-bypass user can't satisfy RLS). v2 therefore **builds on the existing combo model and extends it**, shipping as **one effort with a single bank regeneration** (the owner chose one big release over a phased rollout). Decisions below supersede the PRD-v1 sections noted inline; where they touch combo code they *extend* it rather than rebuild it.

- **Combo-threshold grading (already built; extended here).** The two-piece combo model — always place both pieces; score 0–100 across the combo cross-product; **Correct = score ≥ 95**; no wrong-first short-circuit — landed in #31–#35 and is live. It already resolved the "my answer was identical to #1 but graded wrong" report (the v1 short-circuit had failed the puzzle on a first move that merely differed from the engine's *setup* move). v2 extends it with outcome-by-resulting-board matching (next bullet), tuck/spin combos, difficulty, and dedup — not a rewrite. Supersedes PRD § Checker.
- **Matching is by outcome, not placement tuple.** An attempt matches a stored combo iff it yields the **same resulting board (the locked cells after both pieces)** — not by `(rotation, col)`, not by input path. This is rotation-index- and path-independent, so app-vs-StackRabbit rotation-numbering mismatches can never mis-grade a same-cells answer, and tucks/spins are graded by where they rest. `placement.ts`'s `toPlacement` — which searches only the hard-drop placement space and returns `null` for any tuck — is **removed**; each stored combo carries its placements (for replay) plus a canonical resulting-board key (for matching).
- **Tucks and spins are first-class.** Reversing the PRD's "there is never a tuck answer": the generator enumerates **all collision-reachable resting placements** (hard-drop + tuck + spin) and StackRabbit values each, so a tuck/spin can be the optimal combo. *Binding invariant:* generation's enumerated placement set must be a **superset of what the play-app input can place**, or outcome-matching would wrongly reject a legal tuck. Supersedes PRD §§ Hz-invariance, Checker, and the Out-of-Scope "tuck/spin-required" line.
- **Hz-invariance kept but narrowed.** "Execution speed" now means **left/right traverse speed only** (slow-tap vs fast-DAS reaching distant columns). Tuck/spin *capability* is **granted** — assumed always executable — and is no longer gated; the gate keeps only puzzles whose optimal answer doesn't flip with horizontal movement speed. Expected to *raise* yield, since the old gate over-rejected tuck cases. (Operationalizing this against the live StackRabbit input-timeline API is a generation-tuning task: the timeline knob couples horizontal reach and tuck reachability, so isolating the former likely needs a geometric reachability pass plus an explicit horizontal-reach check, not a two-timeline comparison.)
- **Free-positioning input.** The ghost gains collision-aware move/rotate/**soft-drop** so the player can slide under overhangs and rotate into pockets — no timer, so it stays pure judgment. Replaces the hard-drop-only `legalColumns` model. Supersedes the PRD ghost-input sketch.
- **Difficulty is a generation property.** Computed per puzzle as a **combination of `acceptCount` (combos scoring ≥ 95) and `margin` (gap between the best combo and the best one below 95)** — few acceptable answers and/or a large gap = hard. Stored raw (both signals) and mapped to the puzzle's **seed rating** (harder → higher seed), which bootstraps matchmaking immediately instead of waiting for crowd data. Supersedes PRD § flat-seed / "difficulty not derived from engine signals."
- **Board-health floor relaxed.** The R3 floor kept boards StackRabbit rates *highly* — clean boards where many moves are fine — which biased the bank toward **easy**. It is relaxed to a **fairness/garbage-only** floor; the difficulty target shapes the rest, biased hard with an easy tail kept for low-rated/new players. Supersedes the 2026-06-20 board-health-floor decision.
- **Deduplication (new gate).** Reject a candidate whose **`(piece1, piece2)` match and whose board is within a small Hamming distance** of any puzzle already accepted (checked against the in-progress batch *and* the existing bank). Catches near-identical look-alikes, not just byte-identical duplicates. None existed before.
- **Rating persistence + matchmaking (fixes "rating never changes").** Root cause: the dev-bypass user is an all-zeros UUID with no auth session, so RLS `auth.uid() = user_id` fails and every write is silently swallowed by the `finish()` catch in `PuzzleSession.tsx`; the open-access live site has the same problem for everyone. Fixes: (a) **anonymous Supabase auth for every visitor**, giving a real `auth.uid()` so player ratings persist (per device, portable, linkable to email/Google/Discord later); (b) **puzzle ratings tallied offline in batches** from the `attempts` table (proper Glicko-2 rating periods) — the play app writes only `user_ratings` + `attempts`, puzzles stay read-only to the client; (c) **matchmaking selection** — draw the next puzzle at random from puzzles within a **rating band around the player** (auto-widening if too few), **excluding recently-seen puzzles** (a cooldown — a puzzle can return later, just not soon). One query delivers both "around my rating with variance" and "no repeats for a while." Supersedes PRD § random/sequential selection and the matchmaking Out-of-Scope line.
- **Screenshot submission (new).** A front-end affordance to submit puzzles, **screenshot-only to start**. Solving requires StackRabbit (offline-only, hard guardrail), so the browser only **uploads the image to a submission queue**; the **offline pipeline OCRs the NES grid into board + pieces, solves, runs gates/dedup/difficulty, and banks or rejects** it. Submitted puzzles go live after the next generation run; the engine is never deployed. Extends PRD § board-source pluggability (a screenshot source alongside self-play).
- **Accepted costs.** A **third full bank regeneration** (new puzzle IDs; post-regen attempts orphaned — same pattern accepted twice before). Schema: replace `first_values`/`second_values` with a `combos` top-K column (each combo's placements + 0–100 score + resulting-board key); add `accept_count` / `margin` difficulty columns; add a `submissions` table + a Storage bucket for images; add an RLS path for anonymous sessions. The **replay** must learn to animate tucks (it currently assumes every optimal placement is tuck-free and drops straight down). `checker.ts`, `placement.ts`, the placement enumeration, the input layer, and selection are all rewritten.

### 2026-06-20 — Combo-grading overhaul (grilling session #2)

A second `/grill-with-docs` session reworked grading, feedback, and board generation. These decisions **supersede parts of the earlier same-day UX-overhaul entry** — specifically the "no cross-product needed" premise, the exact-match checker, the strip-plot solutions chart, and the `first_values`/`second_values` model — and require **another full bank regeneration**.

- **Grading → two-piece combo model.** A puzzle becomes "find the best two-piece combo." The player **always places both pieces** (the wrong-first-move short-circuit in `checker.ts`/`PuzzleSession.tsx` is removed). An attempt is the combo `(placement₁, placement₂)`, scored by StackRabbit's evaluation of the board after both placements, field-normalized 0–100 across the puzzle's combos. **Correct = score ≥ 95** (within 5% of the best). The binary solved/failed signal for co-rating follows that bar (a near-miss and a catastrophic miss both count as failed — Glicko-2 stays binary).
- **No cross-product → cross-product, top-K stored.** This reverses the first overhaul's "a wrong first move ends the puzzle, so no cross-product is needed." Generation now sweeps the full cross-product (~400–1000 combos/puzzle) to rank and normalize, but **stores only the top-K (K ≈ 30, tunable)** ranked combos (placements + 0–100 score + total ranked count). Rows stay tiny; no live engine at play time. Player buckets: top-5 → highlighted in the list; ranks 6–K → shown with exact rank+score; beyond K → "too low to rank" (this is R5's "too bad to evaluate" case).
- **Gates.** The discrimination/unambiguity gate is **dropped** — combo-threshold grading needs no unique best. **Hz-invariance is kept**, retargeted from "optimal move per ply" to "the best combo is identical across slow-tap/fast-DAS." Puzzle quality now rests on board quality + the ≥95 bar (accepted risk: an occasional trivially-passable puzzle).
- **Board quality (R3) — StackRabbit board-health floor.** Keep a snapshot only if its **minimum best-move value across all 7 piece types** clears a moderate, tunable floor — a piece-independent proxy for "a board StackRabbit rates highly," since StackRabbit exposes no static board evaluation. A cheap geometric pre-filter (holes/bumpiness) drops obvious garbage first; the floor runs as an early gate before the combo sweep. Chosen over pure geometric thresholds because the owner wanted a StackRabbit-grounded measure; min-over-7 (not the actual current piece) keeps it about the board, not the piece draw; "don't overdo it" → keep the floor moderate to protect yield. Self-play noise may be lowered to feed cleaner candidates.
- **Feedback (R4/R5).** The strip-plot solutions chart is replaced by a **verdict banner** (Correct/Incorrect + 0–100 score) and a **ranked combo list**: top-5 with scores, the player's combo highlighted if among them, otherwise a row below showing rank+score (6–K) or "too low to rank" (beyond K). Rows are **interactive** — selecting one animates that combo on the central board (the existing replay parameterized by `(p1, p2)`); the player's move is selected by default.
- **Colors (R1).** The feedback/replay path never passed `colorGrid` to the board ([Feedback.tsx](../../apps/play/src/feedback/Feedback.tsx), [replay.ts](../../apps/play/src/feedback/replay.ts)), so the stack reverted to white after placement. Fix: make the replay **color-aware** (base = the puzzle's color grid; each dropped piece paints its NES color group) and pass `colorGrid` to the feedback board. The original stack never goes white; every animated piece (player's or engine's) carries its group color.
- **Layout (R6).** The board shipped at `min(40vh, …)` (not the ~80vh the first overhaul intended) under two stacked headers with `align-items: start`, forcing scroll. Fix: a **slim combined top bar** (small wordmark + Play/History/Controls nav, subtitle dropped), board grown to ~`min(72vh, …)` and the play area vertically centered, so it holds on one screen on desktop/tablet (phones still stack/scroll).
- **Accepted costs.** Another full bank regen → new puzzle IDs, the live bank replaced again, post-#27 attempts orphaned (same pattern accepted on the first overhaul). Schema: replace `first_values`/`second_values` with a `combos` column (top-K). `checker.ts` rewritten (no short-circuit; combo lookup + threshold).

### 2026-06-20 — UX overhaul (grilling session): layout, colors, solutions chart, replay, input, history

A `/grill-with-docs` session reworked the play app's UX. The decisions below supersede the relevant feedback/layout sketches in the PRD and are filed as phased GitHub issues (Phase 1 = client-only; Phase 2 = offline bank regeneration + schema; Phase 3 = client features that consume the new data).

- **Performance — drop the CRT layer.** Scroll/general jank traced to the full-viewport `body::after` overlay: `mix-blend-mode: multiply` plus a 160px-blur inset shadow forces a whole-viewport re-composite on every paint. Removing the overlay and the body grid texture for a flat look was chosen over keeping a cheapened effect. The 200-`div` board is kept as-is (a larger board is the same cell count).

Supporting UI flanks it: left = rating + on-screen controls; right = the NES next-box while solving, then the solutions chart after an attempt (they never co-occur). One-screen is guaranteed on desktop/tablet only; phones collapse to a stacked, scrollable column. - **Piece colors — authentic, via full regen.** The existing bank cannot recover per-cell colors (no piece history was stored), so authentic colors require a full bank regeneration with color-tracking self-play — chosen over client-side synthesis. - **Next-piece box.** Rendered as a real piece graphic (spawn orientation, NES color) in a bordered box, top-right (NES-accurate). The current piece stays the on-board ghost (no separate display). The box is empty on placement 2 (no lookahead).
- **Replay animation.** The piece spawns at top-center and performs one eased rotate-and-slide during the upper part of the fall, then drops straight to rest; clip-checked so it never passes through the stack; a flash-and-collapse plays when a line clears; honors `prefers-reduced-motion`. - **Solutions chart — replaces the geometric-metrics table.** After an attempt, feedback shows two per-piece value distributions instead of the holes/bumpiness/height table. Both are precomputed at generation (`first_values` / `second_values`) — no live engine, sidestepping the deferred totalValue-% feature. Each is drawn as a strip plot (dots = alternatives, ★ = optimal, ● = the player's move) with a rank callout ("5th of 17"); positions are field-normalized 0–100 and rank is the headline number. The `boardMetrics` functions remain (the generator still uses them); only the player-facing table is removed.
- **Input — rebindable keys.** Defaults: ←/→ move, `z` = rotate CCW, `x` = rotate CW (this adds counter-clockwise rotation, which did not exist), Enter/Space confirm, ↑ = rotate-CW alias. Five actions are rebindable via a Controls panel (press-to-bind; conflicts warn rather than silently double-bind). Bindings persist in a new Supabase user-prefs table so they sync across devices, like the rating.
- **History view.** A header nav (Play / History / Controls) opens a sortable/filterable list of past attempts (date / difficulty / result), newest-first, paginated. Difficulty is read by joining attempts → puzzles. Each row re-opens the puzzle read-only in the Feedback view; only post-regen attempts are re-openable (older puzzle IDs are gone).
